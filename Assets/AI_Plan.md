# AI Bots Plan for BattleMagi

## Цель
Добавить в игру сервер-авторитативных AI-ботов, которые ведут себя как игроки: перемещаются по карте, кастуют заклинания, подбирают объекты, участвуют в бою и матч-цикле.

## Шаг 1. Архитектура и точки интеграции
- Зафиксировать точки интеграции в текущей системе:
  - `Assets/Scripts/PlayerSpawner.cs`
  - `Assets/Scripts/net/PlayerManager.cs`
  - `Assets/Scripts/net/TeamManager.cs`
  - `Assets/Scripts/core/PlayerDamageable.cs`
- Определить модель участника-бота (`BotParticipant`): `BotId`, команда, архетип, состояние, текущая цель.
- Развести понятия `BotId` и `OwnerClientId`, чтобы не ломать текущие сетевые пайплайны.

### Критерий готовности
- Понятен источник правды для ботов (сервер), схема идентификаторов и места подключения в матч-цикле.

## Шаг 2. Серверный жизненный цикл бота (MVP)
- Реализовать менеджер ботов:
  - спавн
  - смерть
  - респавн
  - удаление в конце матча
- Добавить регистрацию ботов в учет участников матча и таблицу счета.
- Обеспечить совместимость с текущей логикой завершения матча.

### Критерий готовности
- Бот стабильно появляется/умирает/возрождается и корректно учитывается в матче.

## Шаг 3. Передвижение и навигация (MVP)
- Подготовить NavMesh для целевых сцен (`com.unity.ai.navigation` уже есть в проекте).
- Добавить `BotMovementController`:
  - выбор точки интереса/waypoint
  - обход препятствий
  - логика anti-stuck
- Согласовать с ограничениями текущего перемещения (`Assets/Player/Scripts/FirstPersonMovement.cs`).

### Критерий готовности
- Бот уверенно перемещается по карте без частых застреваний и сетевого дрожания.

## Шаг 3.5. Прыжки и перепады высот (MVP)
- Доработать `BotMovement`/`BotMovementController` для работы с вертикальными перепадами:
  - запрыгивание на низкие препятствия (камни)
  - спрыгивание с уступов/гор по маршруту
  - кулдаун и recovery для повторных попыток прыжка
- Согласовать с текущей физикой (`PlayerPhysics`) и анимацией (`BotAnimator`).

### Критерий готовности
- Бот стабильно преодолевает низкие препятствия прыжком и не застревает при спусках с высоты в типовых зонах карты.

## Шаг 4. Бой и каст заклинаний (MVP)
- Добавить `BotCombatController`:
  - поиск цели (дистанция, line of sight, приоритет)
  - выбор заклинания (дальность, cooldown, ресурс)
  - момент каста
- Вызывать существующий API `SpellCasterPlayer` (не эмулировать ручной ввод).
- Учитывать команды через `TeamManager.AreEnemies`.

### Критерий готовности
- Бот регулярно и валидно кастует по врагам, урон/киллы корректно учитываются.

## Шаг 5. Подбор объектов и поведение выживания (MVP)
- Добавить `BotObjectiveController` с приоритетами:
  - если мало HP/ресурсов -> искать pickup
  - иначе -> бой/позиционирование
- Интегрировать подбор через текущие `PickUp`/`PickUpSpawn`.
- Добавить fallback, если объект недостижим.

### Критерий готовности
- Бот осмысленно подбирает объекты и это повышает его выживаемость.

## Шаг 6. Расширение после MVP
- Поддержка режимов и ролей:
  - FFA
  - Team
  - CTF
- Профили сложности:
  - скорость реакции
  - агрессия
  - точность
  - частота смены решений
- Улучшенная тактика: удержание дистанции, фокус целей, отступление.

### Критерий готовности
- Боты играют адекватно в разных режимах и заметно отличаются по сложности.

## Порядок внедрения
1. MVP-1: серверный спавн + регистрация в матче + базовое перемещение.
2. MVP-2: бой (таргетинг + каст) + базовый подбор объектов.
3. MVP-3: полная совместимость с респавном/score/концом матча.
4. Expansion-1: режимы Team/CTF.
5. Expansion-2: сложность, роли, оптимизация.

## Зависимости
1. `PlayerManager`/`TeamManager`/`PlayerSpawner` должны поддержать виртуальных участников.
2. `PlayerDamageable` и kill/assist pipeline должны корректно работать с ботами.
3. Сцены должны иметь поддерживаемый NavMesh-пайплайн.

## Definition of Done
1. Бот в сетевом матче стабильно двигается, кастует, подбирает, умирает и респавнится.
2. Scoreboard/killfeed/условия победы корректны для людей и ботов.
3. Нет критичных сетевых рассинхронов при 4-8 ботах в типовом матче.
4. Параметры поведения настраиваются без изменения кода (через профиль/ScriptableObject).

## Риски и меры
1. Риск: жесткая завязка на `OwnerClientId`.
   - Мера: адаптерный слой с явным `BotId`.
2. Риск: рост сетевой нагрузки.
   - Мера: ограничить частоту решений/переключений целей/кастов.
3. Риск: сложные зоны навигации.
   - Мера: recovery-перепланирование и blacklist недостижимых точек.
4. Риск: дисбаланс сложности.
   - Мера: профили поведения + метрики эффективности (K/D, частота каста, value от pickup).
